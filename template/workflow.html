Workflow

overview

  1. get the data
    • facsimiles
      • link to existing facsimiles, preferably through IIIF
      • scan your physical artefacts*
    • text
      • scrape existing transcriptions
      • use Transkribus, eScripta/eScriptorium for HTR (with our model trained on 1000+ pages from the OpenArabicPE corpus)
  2. save and share the data
    • facsimiles: Internet Archive (supports IIIF)
    • working copy of everything else: GitHub
    • longterm preservation: Zenodo
      • make sure to add ORCIDs for all contributors
  1. edit the data
    • support for RTL in free text editors varies between OS
    • use open, publicly-funded (web)editors, such as TextGrid Lab
    • make use of version control and stable IDs (e.g. ORCID) for transparent authorship attribution
  2. disseminate the data
    • generate static webviews:
      • XSLT1 to render XML files in the client’s web browser
      • GitHub actions to generate static webpages
    • host webviews, project blogs: gh-pages
    • bibliographic database: Zotero
    • full-text search: Google’s programmable search engines

Resulting workflow of OpenArabicPE

1. get the data

  • facsimiles
    • link to existing facsimiles from British Library’s “Endangered Archives Programme” (EAP), Translatio Bonn, Arshīf al-majallāt […] al-ʿarabiyya etc., preferably through IIIF
    • scan/ photograph your physical artefacts (at the lowest sustainable resolution)
  • text
    • scrape existing transcriptions from shamela.ws, et al.
    • use Transkribus, eScripta/eScriptorium for HTR (with our model trained on 1000+ pages from the OpenArabicPE corpus)

2. model the data

Structure the text string into issues, sections, articles with bylines …

  • widely accepted standard for textual editions: Text Encoding Initiative (TEI/XML)
    • active community
    • pre-requisite for grant funding
    • easy to archive (XML = plain text)
  • re-use / adapt domain specific encoding schemas within the TEI
  • try to script basic modelling using patterns in your source text:
    • regular expressions
    • XSLT, Python, R, whatever you are most comfortable with
  • automatically model derivative bibliographic data: MODS, METS, BibTeX, …

  • why TEI?
    • I knew it already
    • widely adopted standard in the digital editing world
    • necessary for grant-funding in the Global North
  • why not TEI
    • steep learning curve
    • not particularly well-suited to Arabic texts?

3. edit the data

  • make use of version control for transparent authorship attribution and damage control:
    • .git is open source and available for all OSs
  • plain-text (including XML) editors
    • should be syntax aware
    • RTL: support in text editors is a mixed bag
  • XML editors proper
    • should be schema aware to validate the encoding
    • RTL:
      1. oXygen XML editor (99 USD) allows to separate content and tags
      2. Visual Studio Code (free) with the right extensions is a viable second
  • editing tools depend on modelling decisions and file formats

4. save and share the data

  • facsimiles: Internet Archive (supports IIIF)
  • working copy of everything else: distributed version control platforms (GitHub, GitLab
  • longterm preservation: publicly-funded, open repository (Zenodo)
    • make sure to add ORCIDs for all contributors
    • provides versioned DOIs
  • authority data (people, titles, etc.): Wikidata
  • provide suitable open licenses for re-use: Creative Commons, MIT, Public Domain (CC0)

5. present the data

  • Hosting: GitHub Pages can expose your data repository to the web
  • Generate static webviews
    • removes need for backend and minimises traffic
    • easy to archive
    • on the fly: XSLT1 (TEI Boilerplate) or JS (CETEICEan) to render XML files in the client’s web browser
    • pre-computed: GitHub actions give access to virtual machines
  • bibliographic database: Zotero group
    • mitigates against the absent backend
    • browse and search independent of file structure
  • full-text search across the entire corpus: Google’s programmable search engine
